# Задача M2.1 — Чанкер текста на страницы

**Размер.** S (~1 день)
**Зависимости.** M1.1–M1.5.
**Что строится поверх.** M3.1 (seed-пайплайн), M8.1 (persistence) сохраняют в БД уже нарезанные страницы.

---

## О проекте (контекст)

**en-reader** — веб-читалка. Книги могут быть очень большими (100 000+ слов), отдавать их одной страницей на фронт — убивает Safari и съедает память. Нужна пагинация ещё на бэке: книга режется на страницы 100–1000 английских слов **по границам предложений** (не в середине).

100 слов — минимум «страницы» (меньше — визуально неудобно). 1000 — верх (больше — тяжело на фронте). Большинство страниц попадёт в 300–600.

Предложения не режем — читатель не должен видеть оборванное «…the man who came».

---

## Что нужно сделать

Функция, принимающая размеченный текст (tokens + units) и возвращающая список `Page`.

---

## Что входит

### 1. Dataclass `Page`

В `src/en_reader/models.py`:
```
@dataclass
class Page:
    page_index: int
    text: str              # кусок оригинального текста для этой страницы
    tokens: list[Token]    # копия токенов, чьи idx_in_text попадают в диапазон страницы,
                           #   с пересчитанным idx_in_text относительно начала страницы
    units: list[Unit]      # Unit, все token_ids которых попали в эту страницу
```

### 2. Функция `chunk(tokens, units, full_text) -> list[Page]`

В `src/en_reader/chunker.py`.

**Алгоритм:**
1. Проходим предложения по порядку (используем `is_sent_start` как границу).
2. Для каждого предложения считаем число «слов» — это число токенов с `pos in {"VERB", "NOUN", "ADJ", "ADV", "PROPN", "PRON", "DET", "ADP", "AUX", "NUM", "INTJ"}` (т.е. всё кроме PUNCT, SPACE, SYM, X). Это простая прокси-метрика английских слов.
3. Накапливаем предложения в «буфер страницы», пока слов в буфере < 100.
4. Когда слов ≥ 100 — проверяем следующее предложение. Если его добавление превысит 1000 слов — закрываем страницу до него. Иначе — добавляем и проверяем дальше.
5. Специальный случай: если **одно** предложение > 1000 слов (редкий артефакт парсинга) — оно становится самостоятельной страницей, с warning в лог.

### 3. Формат `Page.text`

- `text` = подстрока исходного текста от начала первого до конца последнего токена страницы, **rstripped**.
- Между страницами при последующей сериализации (JSON → БД) подразумевается разделитель `\n\n`, но в `Page.text` его не включаем.

### 4. Пересчёт `token.idx_in_text` и `unit.token_ids`

- `token.idx_in_text` в Page — относительный: char offset от начала `page.text`.
- `unit.token_ids` — индексы токенов **в списке `page.tokens`**, не глобальные.
- Это нужно, чтобы страницы были самодостаточны; фронт получает страницу и рендерит без ссылок на глобальную нумерацию.

### 5. Обработка units на границах страниц

- Если Unit целиком в пределах одной страницы — нормальный случай.
- Если Unit разрезан границей страницы (не должен, мы режем по границам предложений, но split PV технически может) — **не создаём Unit**, логируем warning. В логике: если все `token_ids` Unit в одной странице — включаем; иначе — дропаем Unit с логом.
- Split PV обычно внутри одного предложения → внутри одной страницы. Если нет — это редкий кейс, дропаем.

### 6. Логирование

- При создании страницы > 1000 слов (из-за гигантского предложения) — `logger.warning("Oversized page: %d words", n)`.
- При дропе split PV через границу страницы — `logger.warning("Dropped split PV across page boundary")`.

### 7. Тесты `tests/test_chunker.py`

- **Маленький текст.** 50 слов → одна страница с 50 словами.
- **Граница 100.** Текст из нескольких коротких предложений, где первая страница набирается после 5-го предложения. Проверить, что страница закрывается на границе предложения.
- **Несколько страниц.** Текст 2500 слов → 3–4 страницы, каждая 100–1000 слов, кроме последней (может быть < 100).
- **Гигантское предложение.** Предложение из 1200 слов — страница из одного предложения, warning в логе.
- **Инвариант восстановления.** `"\n\n".join(p.text for p in pages) == full_text.rstrip()`.
- **Инвариант границ.** Каждая страница начинается с токена с `is_sent_start=True`.
- **Инвариант отсутствия разрывов.** Для каждого предложения в исходном тексте: все его токены оказались ровно в одной странице.

## Технические детали и ловушки

- **Что считать «словом»**: простая прокси `pos ∈ whitelist` достаточна. Не усложняй — это не про точный word-count, а про грубый таргет «страница размером с книжную».
- **Последняя страница** может быть < 100 слов — это норма. Специально не накачивай её до 100.
- **Пробельные токены.** spaCy в en_core_web_sm не производит SPACE-токенов как отдельные — пробелы между токенами сидят в `token.whitespace_`. Значит, `page.text` можно собрать как `"".join(tok.text + tok.whitespace_)`, это даст точную подстроку.
- **Rstrip финальной страницы** — да. Rstrip последней — тоже да. Остальные — внутри страницы не трогаем (сохраняем trailing whitespace предложения).
- **`is_sent_start` на первом токене страницы** — всегда True, потому что режем по границам.
