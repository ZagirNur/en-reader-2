# Задача M1.3 — Детекция устойчивых выражений (MWE)

**Размер.** S (~1 день)
**Зависимости.** M1.1, M1.2.
**Что строится поверх.** M1.4 (phrasal verbs; MWE имеет приоритет над phrasal), M1.5 (golden-тесты включают MWE).

---

## О проекте (контекст)

**en-reader** — веб-читалка, где «переводимые» токены подчёркнуты. Без MWE-поддержки каждое слово в `in order to` подчёркивалось бы отдельно, и пользователь получал бы три отдельных перевода вместо одного связного. Задача — собрать курируемый словарь устойчивых выражений и объединять их в **один** Unit.

---

## Что нужно сделать

Реализовать MWE-матчинг: в списке токенов находить устойчивые выражения из курируемого словаря, создавать на каждый матч один `Unit kind="mwe"`, а токенам этого выражения проставить `unit_id` и снять одиночный `translatable`.

---

## Что входит

### 1. Файл `data/mwe.txt`

Курируемый список устойчивых выражений, ~300–500 записей. По одному выражению на строку, нижний регистр, пробел-разделённые леммы или слова.

Категории:
- Составные наречия/предлоги: `in order to`, `as well as`, `by the way`, `on the other hand`, `at the same time`, `in fact`, `in addition`, `in spite of`, `instead of`, `according to`, `due to`, `as a result`, `as if`, `as though`, `even though`, `as long as`, `as soon as`, `in front of`, `next to`, `out of`, `because of`.
- Фиксированные идиомы среднего уровня: `make sense`, `take care`, `for the sake of`, `at least`, `at most`, `at last`, `all of a sudden`, `by no means`, `for instance`, `for example`.
- Связки с глаголами, которые не phrasal verbs (для phrasal — свой словарь в 1.4).

Формат: `in order to`, `as well as`. Не больше одного выражения на строку, комментарии `#...`.

### 2. Загрузка словаря

- `_load_mwe() -> list[list[str]]` — читает `data/mwe.txt`, splits на слова, возвращает список списков. Ленивый singleton.

### 3. MWE-матчер через spaCy `PhraseMatcher`

- В `nlp.py` добавить функцию `mark_mwe(doc: spacy.tokens.Doc, tokens: list[Token], units: list[Unit]) -> None` (или интегрировать в основной пайплайн).
- Использовать `spacy.matcher.PhraseMatcher(nlp.vocab, attr="LEMMA")` — матчинг по леммам, чтобы `makes sense` тоже матчился как `make sense`.
- Для каждого выражения — добавить шаблон (`nlp("in order to")`).
- При матче:
  - Создать `Unit(id=next_id, token_ids=[...индексы], lemma="in order to", kind="mwe")`.
  - У всех входящих токенов: `unit_id = Unit.id`, `translatable = True` (Unit целиком translatable; одиночные токены смысла уже не имеют, но свойство `translatable` на Unit определяется через Unit, не через токены — см. ниже).
- Перекрывающиеся матчи: **greedy longest first**. Если два MWE конкурируют — выигрывает тот, что длиннее. При равенстве длины — первый по позиции.

### 4. Концептуальное уточнение: `translatable` на Unit

После интеграции MWE правило «что подчёркивается» на фронте: токен подчёркивается, если он входит в Unit (`unit_id != None`), **или** сам `translatable=True` и `unit_id=None` (одиночное слово).

Чтобы это не расходилось: оставь у токенов внутри MWE флаг `translatable=True` (не сбрасывай в False). Фронт будет смотреть на Unit как на целое. Это удобно, хотя технически можно было бы и обнулять. Единственное правило: **одиночный токен, поглощённый Unit, не должен порождать отдельный рендер** — на фронте это решается группировкой по `unit_id` (в задачах M3.3).

### 5. Функция сборки единиц

Нужно чтобы `tokenize()` (или главная функция разметки, которая соберёт всё вместе) возвращала не только `tokens`, но и `units`. Вероятная сигнатура:
```
def analyze(text: str) -> tuple[list[Token], list[Unit]]:
    ...
```
В задаче 1.1 был только `tokenize`. Сейчас заведи `analyze()` как orchestration-функцию:
1. `doc = get_nlp()(text)`.
2. Построить `tokens` (как в 1.1).
3. Проставить `translatable` (логика из 1.2) — пока для Unit=None токенов.
4. Прогнать MWE → создать Units, проставить `unit_id`.
5. Вернуть `(tokens, units)`.

Старая `tokenize()` может остаться как частный случай (без MWE), но все будущие задачи будут звать `analyze()`.

### 6. Тесты `tests/test_mwe.py`

- `"She spoke as well as anyone."` — `as well as` распознан как один Unit kind=mwe с token_ids=[1,2,3] (индексы токенов `as well as`).
- `"In order to win, he worked hard."` — `in order to` → Unit.
- Перекрытие: `"in fact in order to"` — два Unit без пересечений.
- Инварианты: ни один токен не входит в два Unit; все `token.unit_id` валидны.

## Технические детали и ловушки

- **`PhraseMatcher(attr="LEMMA")`** матчит по леммам — это то что нужно. `attr="LOWER"` был бы строгим матчем по тексту, что ломает `makes sense` vs `made sense`.
- **Шаблоны создавай через `nlp.make_doc`**, а не `nlp()` (полный пайплайн тяжёлый): `patterns = [nlp.make_doc(x) for x in phrases]`. Но `make_doc` даёт токены БЕЗ леммы. Для matcher по LEMMA нужен полный пайплайн. Решение: один раз прогнать все фразы через `nlp.pipe(phrases)` на старте.
- **Greedy longest**: spaCy `PhraseMatcher` не делает это сам. Получи все матчи, отсортируй по `(-length, start)`, проходи и пропускай перекрывающиеся.
- **Помни про регистр**: lemma spaCy обычно lowercase, но проверь на именах собственных.
