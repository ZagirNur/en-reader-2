# Задача M1.1 — Скелет проекта и базовый NLP-пайплайн

**Размер.** M (~2 дня работы одного разработчика)
**Зависимости.** Нет — это стартовая задача проекта.
**Что строится поверх этой задачи.** 1.2 (STOP_WORDS + правило translatable), 1.3 (MWE), 1.4 (фразовые глаголы), 2.1 (chunker), 3.1 (seed-пайплайн), 12.x (парсеры) — все они импортируют твои `Token`, `Unit` и `tokenize()`.

---

## О проекте (тебе для контекста)

**en-reader** — веб-читалка английских книг для русскоязычных учащихся уровня B1–C1.

Как это выглядит для пользователя: открыл книгу — видит текст, в нём подчёркнуты «достойные перевода» слова (редкие существительные, глаголы, прилагательные, фразовые глаголы). Кликнул по слову — прямо в строке рядом с английским появляется русский перевод (inline, не всплывашка). Переведённое раз слово система запоминает на уровне аккаунта и подсвечивает его во всех будущих книгах того же пользователя.

Как это устроено под капотом:
1. **При загрузке книги** — один проход детерминированной NLP-разметки на spaCy: считаются токены, границы предложений, определяется «что переводимо», находятся устойчивые выражения и фразовые глаголы.
2. **При клике пользователя** — один вызов LLM (Google Gemini), которой даётся одно английское слово и его предложение; возвращается одна русская строка.
3. **Словарь пользователя** — SQLite-таблица `(user_id, lemma, translation)`. Повторный клик по известному слову не зовёт LLM, а берёт перевод из словаря.

**Твоя задача** — самый фундамент. Ты делаешь токенизатор и скелет проекта, на которых потом вырастут все остальные NLP-правила. Никаких бизнес-фич ещё нет, ни фронта, ни БД.

---

## Что нужно сделать

Поднять python-проект и собрать минимальный NLP-конвейер:
**вход** — строка английского текста → **выход** — список объектов `Token` с атрибутами и информацией о границах предложений.

Плюс — типы `Token` и `Unit` с полями, которые нужны следующим задачам, даже если в этой ты заполняешь не все.

### 2. Dataclasses в `src/en_reader/models.py`

```
Token:
  text: str              # исходный текст токена (как в строке)
  lemma: str             # нормальная форма
  pos: str               # POS-тег (spaCy coarse: NOUN, VERB, AUX, ADJ, ...)
  is_sent_start: bool    # True, если токен начинает новое предложение
  idx_in_text: int       # char offset от начала входной строки
  translatable: bool = False     # считает задача 1.2
  unit_id: int | None = None     # считают задачи 1.3 / 1.4
  pair_id: int | None = None     # считает задача 1.4 (split phrasal verb)

Unit:
  id: int
  token_ids: list[int]
  lemma: str
  kind: str                      # ожидаемые значения: word|mwe|phrasal|split_phrasal
  is_split_pv: bool = False
  pair_id: int | None = None
```

В этой задаче ты **не создаёшь** Unit-объекты — только описываешь тип. Создавать будут задачи 1.3 и 1.4.

### 3. Обёртка над spaCy в `src/en_reader/nlp.py`

- Функция `get_nlp() -> spacy.Language`. Возвращает singleton модели `en_core_web_sm`, **ленивая** загрузка при первом вызове (`_nlp: spacy.Language | None` на уровне модуля).
- Почему singleton: загрузка модели ~1–2 с, а она будет вызываться в тестах и из следующих задач многократно.

### 4. Функция `tokenize(text: str) -> list[Token]`

- Прогоняет текст через spaCy (`nlp = get_nlp(); doc = nlp(text)`).
- Для каждого spaCy-токена создаёт `Token` со значениями:
  - `text = tok.text`
  - `lemma = tok.lemma_`
  - `pos = tok.pos_`
  - `is_sent_start = bool(tok.is_sent_start)` (нормализуй `None` → `False`)
  - `idx_in_text = tok.idx`
- Остальные поля — дефолты.
- Включая пунктуацию и пробелы — никакой фильтрации, токенизация без потерь.

### 5. Тесты в `tests/test_tokenizer.py`

- **Базовая токенизация.** `tokenize("The cat sat on the mat.")` возвращает 7 токенов (`The`, `cat`, `sat`, `on`, `the`, `mat`, `.`). У первого `is_sent_start == True`.
- **Границы предложений.** `tokenize("First sentence. Second sentence.")` — ровно два токена с `is_sent_start == True`, и это первые слова каждого предложения (`First`, `Second`).
- **Инвариант «concat восстанавливает текст».** На фикстуре `tests/fixtures/demo.txt` (абзац ~200 слов английского текста, любой публичный) проход по токенам с учётом зазоров восстанавливает исходную строку посимвольно.
  - Формула зазора: `text[prev.idx_in_text + len(prev.text) : curr.idx_in_text]`.
  - Перед первым токеном — `text[:first.idx_in_text]`, после последнего — `text[last.idx_in_text + len(last.text):]`.

## Технические детали и ловушки

- **spaCy модель ставится отдельно.** `pip install spacy` поднимает только библиотеку; модель `en_core_web_sm` качается командой `python -m spacy download en_core_web_sm` (~15 МБ). Не пытайся ставить её через pip-extra.
- **Парсер в модели уже включён.** `spacy.load("en_core_web_sm")` по умолчанию включает tagger, parser, attribute_ruler, lemmatizer. Отключать ничего не нужно.
- **`is_sent_start` может быть `None`.** spaCy иногда не проставляет явно — приведи к bool.
- **Не используй `nlp.pipe`.** Один вызов `nlp(text)` на текст. Батчинг будет в других задачах.
- **Никакой фильтрации.** Пунктуация, пробельные токены — всё включено. `translatable` везде `False` — его пересчитает задача 1.2 на основе POS и STOP_WORDS.
- **Singleton потокобезопасный не нужен.** Приложение однопроцессное, sync-ассерты достаточно.
