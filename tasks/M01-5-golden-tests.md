# Задача M1.5 — Golden-тесты разметки

**Размер.** S (~1 день)
**Зависимости.** M1.1–M1.4.
**Что строится поверх.** Страховочная сеть для всех будущих изменений NLP. Если после M6/M7 кто-то случайно сломает разметку — golden-тест падает с читаемым diff.

---

## О проекте (контекст)

**en-reader** — веб-читалка. Разметка «что переводимо, что в составе MWE, что phrasal» — ядро продукта. Без регрессионной страховки она тихо поедет: обновление spaCy или словаря легко ломает поведение на существующих книгах.

Решение — набор эталонных фрагментов с заранее зафиксированной разметкой. После любого изменения пайплайна прогоняем golden-тесты; любое непреднамеренное расхождение — красный тест с diff.

---

## Что нужно сделать

Зафиксировать ручную разметку 3–5 эталонных абзацев + автоматизировать её сверку в тестах, включая режим обновления эталона.

---

## Что входит

### 1. Эталонные фрагменты

В `tests/fixtures/golden/` положить:
- `01-simple.txt` (~50–100 слов обычного нарратива).
- `02-phrasal.txt` (~100 слов с несколькими contiguous и split PV).
- `03-mwe.txt` (~100 слов с несколькими MWE, включая составные предлоги).
- `04-dialogue.txt` (~100 слов диалога с тире/кавычками).
- `05-complex.txt` (~150 слов смешанного типа с пунктуацией).

Тексты брать из публичных источников (Project Gutenberg). Указать источник в начале файла в комментарии, если лицензия требует. На безопасной стороне — писать самому.

### 2. Эталонные JSON рядом

Для каждого `NN-name.txt` — `NN-name.golden.json` с точной структурой:
```
{
  "tokens": [
    {
      "text": "The", "lemma": "the", "pos": "DET",
      "is_sent_start": true, "idx_in_text": 0,
      "translatable": false, "unit_id": null, "pair_id": null
    },
    ...
  ],
  "units": [
    {
      "id": 0, "token_ids": [12, 13],
      "lemma": "look up", "kind": "phrasal",
      "is_split_pv": false, "pair_id": null
    },
    ...
  ]
}
```

Эти JSON-ы **ручной разметки нет**: ты один раз генерируешь их через текущий пайплайн, вручную проверяешь глазами, коммитишь. Дальше они — истина.

### 3. Тест-раннер `tests/test_golden.py`

- Для каждой пары `(txt, golden.json)`:
  1. Прочитать текст.
  2. `analyze(text)` → `(tokens, units)`.
  3. Сериализовать результат в тот же формат.
  4. Сравнить с эталоном.
- При расхождении — вывести `assert actual == expected` (pytest сам покажет читаемый diff).
- Тест параметризуй через `@pytest.mark.parametrize` по всем парам файлов в `golden/`.

### 4. Режим обновления

Флаг `--update-golden` для pytest (`conftest.py`):
```
def pytest_addoption(parser):
    parser.addoption("--update-golden", action="store_true", default=False)
```

В тесте:
```
if request.config.getoption("--update-golden"):
    write(expected_path, actual_json)
    pytest.skip("golden updated")
else:
    assert actual == expected
```

Использование при намеренной смене поведения: `pytest tests/test_golden.py --update-golden` → тесты перезаписывают `.golden.json` → ты руками проверяешь diff в git → коммитишь.

### 5. Инварианты на большом тексте

`tests/test_invariants.py` — отдельный файл:
- Фикстура `tests/fixtures/long.txt` (~10 000 слов любой публичной книги).
- Прогон `analyze(long_text)`.
- Проверки:
  1. `concat(token.text + gap) == original_text` посимвольно.
  2. Ни один token не входит в два Unit: множество `{token_id}` всех Unit не пересекается.
  3. Каждое предложение начинается ровно с одного токена с `is_sent_start=True`.
  4. Все `pair_id` — у пар split_phrasal Unit (ровно 2 Unit на каждый pair_id).

Этот тест отдельно от golden — потому что мы не фиксируем точную разметку 10 000 слов, но хотим проверять свойства.

## Технические детали и ловушки

- **Стабильность между запусками.** spaCy при одинаковом вводе даёт одинаковый вывод (модель детерминированная). Но при обновлении версии spaCy/модели разметка может поехать. Зафиксируй версии в `pyproject.toml` строго (`spacy==3.7.x`, модель — через spaCy download, тоже фиксируется).
- **Сравнение dataclass-ов.** Если `Token` и `Unit` — `@dataclass`, сравнивай через `asdict(token)` vs dict из JSON. Проще: сериализуй через `json.dumps(asdict(tok), sort_keys=True)` и сравнивай строки, или делай глубокое сравнение через pytest.
- **JSON-форматтер.** Используй `json.dumps(..., indent=2, ensure_ascii=False)` при обновлении golden — diff в git читаемый.
- **Порядок Units.** Отсортируй по `(token_ids[0], id)` перед сериализацией — иначе порядок может плавать.

## Что НЕ делать

- Не придумывай «умные» golden-форматы с фаззи-сравнением — строгий JSON-equality.
- Не коммить обновлённые golden без ручной проверки: они должны отражать осознанное изменение поведения, а не «красный тест почему-то, обновим и забудем».
